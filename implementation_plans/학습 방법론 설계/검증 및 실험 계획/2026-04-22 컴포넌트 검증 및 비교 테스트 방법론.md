### 0.3 핵심 원칙 (작성 시 준수)

1. **1부 = sanity, 2부 = comparison** — 두 역할 절대 혼동 금지
2. **2부 모든 항목 = baseline 기준 + config 한 항목 변경** — multi-factor 변경은 별도 sweep
3. **MLflow 일괄 관리** — experiment / run / tag 규칙 통일
4. **로드맵 (`2026-04-20 모듈화된 코드 구현 로드맵.md`) 의 모든 config item 이 2부 TR-C 에 1:1 매핑** — 누락 발견 시 즉시 추가
5. **모든 TR-C variant 마다 sanity check 1회 실행 필수** — Level 별 선택적 실행 정책 (§3.6 참조)
6. **(초안) 의 잔여물** — 의미 있는 것만 흡수 (TR-R-07 → TR-C-PL-NN, TR-X-01~08 → TR-C-MS/PL/TR-NN), 중복/불필요 항목 폐기

### 0.4 Karpathy Recipe 필터링 결과 (§3 작성 시 적용)

**유지**:
- L1 코드 정합성 5항목 (shape, NaN, loss@init, 1/4 대칭, 5D axis)
- L2 학습 동역학 4항목 (overfit one batch, dumb baseline, loss magnitude inventory, gradient norm)
- L3 물리 검증 — **전부 기존 V&V + piecewise-test 자산 reference, 신규 항목 없음**
- L4 Phase 전환 3항목 (1a→1b annealing, 2a/2b unfreeze, 3 K curriculum)

**제외**:
- Visualize before/during predict — 시각화는 §5 모니터링 / MLflow 로 이관
- Backprop chart dependencies — 5D 텐서 batch leakage 검증 비용 vs 이익 낮음
- Generalize special case — 본 프로젝트 노심 dynamics 라 special case 정의 모호
- Phase 2b 중반 ReLoBRaLo activation 3조건 — sanity 가 아닌 trigger 충족 평가 → §6 TR-P-2b 로 분리
- Init well (Xavier/He) — TF2 기본 initializer 적절, 자명
- L_Bateman / L_diff_rel / 2-group scattering / k_eff 신규 sanity — 모두 기존 V&V 또는 L_data scalar 처리로 별도 sanity 항목 불필요
