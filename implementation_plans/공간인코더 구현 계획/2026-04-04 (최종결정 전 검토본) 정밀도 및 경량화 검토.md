# 정밀도 및 경량화 검토

> **작성일**: 2026-04-04
> **개정일**: 2026-04-09 — §4 Attention 비교표를 N=500 (quarter) → **N=720 (halo expand 후)** 으로 갱신. halo (6,6) 결정 (`05_symmetry_mode.md`) 반영
> **배경**: float32 데이터의 수치 정밀도를 유지하면서 모델을 경량화할 수 있는지 검토
> **핵심 결론**: **FP32 사용.** MASTER 원본 유효숫자(4~6자리) < FP32 유효숫자(7.2자리) → 모든 물리량이 FP32 안에서 손실 없음. 경량화는 정밀도 축소가 아닌 구조적 효율화로 달성.

| 포맷 | 순전파(정규화 후) | Physical Loss 계산 | 판정 |

| **FP32** | ✅ | ✅ | **기본 선택** |
| **INT8** | — | — | **부적합** (회귀 출력 정밀도 부족, 핵/열수력 사례 없음) |

### 참고 논문: "FP64 is All You Need" (NeurIPS 2025)

> arXiv 2505.10949 — PINN에서 FP32조차 L-BFGS 수렴을 저해하는 사례 보고. 
> 대류, 반응, 파동, Allen-Cahn 방정식에서 FP32로 학습 실패 → FP64로 해결.

우리 모델은 PINN(PDE를 직접 풀이)이 아닌 **GT 데이터 기반 학습 + physical loss 보조**이므로 상황이 다름. 그러나 physical loss 항(L_Bateman, L_diffusion)의 중간 계산에서 정밀도가 중요하다는 경고.

## 2. TF32 참고 — GPU 내부 가속 (별도 포맷이 아님)

TF32는 사용자가 선택하는 데이터 포맷이 아님. NVIDIA Ampere+ GPU가 **FP32 연산을 내부적으로 가속**하는 모드:
- 코드에서는 100% FP32 → GPU 내부에서 matmul 시 가수를 23bit→10bit로 근사하여 Tensor Core 가속 → 결과는 FP32로 반환
- 사용자가 신경 쓸 것 없음. "FP32를 쓴다"가 답이고, GPU가 알아서 빠르게 처리함

## 3. Mixed Precision (AMP) — 조건부 적용 가능

**전략**: 신경망 forward/backward는 FP16, physical loss 계산은 FP32 유지.

```python
# TF 2.14 Mixed Precision 패턴
tf.keras.mixed_precision.set_global_policy('mixed_float16')

# Physical loss 내부에서만 FP32 강제
def bateman_loss(y_pred, y_true, ...):
    y_pred = tf.cast(y_pred, tf.float32)  # ← FP32로 캐스트
    y_true = tf.cast(y_true, tf.float32)
    # ODE 적분, 반응률 계산 등은 FP32로 수행
    ...
```

**기대 효과**: matmul/attention 연산 ~1.5-2× 속도 향상 (FP16 Tensor Core 활용)
**위험**: softmax, exp 등에서 FP16 overflow 가능 → 표준 AMP가 이를 자동 처리하나, custom ops 주의

**판단**: 학습 속도가 병목이 되면 도입 검토. 초기에는 FP32 순수 학습으로 baseline 확보 후.

## 4. 실질적 경량화 — 구조적 효율화

정밀도를 낮추는 대신, **동일 정밀도에서 연산량을 줄이는** 방향이 적합.

### Attention 방식별 연산량 비교

기존 Voxel Attention 모델을 기준으로:

```
기존 Voxel Attention (CustomMaxViT3D):
  입력: (B, 26, 7, 7) — BoundaryPad 후
  Block Attn: block_size=(4,2,2), 토큰=16, 윈도우=112 → 28.7K/stage
  Grid Attn:  grid_size=(4,2,2), 토큰=8,  윈도우=208 → 13.3K/stage
  1 stage = 42K, 3 stages = 126K
  (단일 타임스텝, 인코더만)
```

| 방식 | Attention ops / stage | 3 stages | ×576스텝 ×2(인코더+디코더) |
|------|---:|---:|---:|
| **Block+Grid** (기존, N=500 quarter 가정) | 42K | 126K | **145M** |
| **Full Attention** (N=500 quarter) | 250K | 750K | **864M** |
| **Full Attention** (N=720 halo) — 개정 | 518K | 1,555K | **1,792M** |
| Full(720) / Block+Grid 비율 | 12.3× | 12.3× | **12.3×** |

> **개정 (2026-04-09)**: halo expand 결정 (`05_symmetry_mode.md`) 으로 인코더 본체는 N=720 (halo 6×6) 처리. (720/500)² = 2.07× FLOPs 증가. 절대치는 여전히 trivial (현재 격자 작아 무시 가능). Full-core (N=1620) 대비 (1620/720)² = 5.06× 더 무거우므로 halo (720) 는 충분히 감수 가능한 trade-off.

576 타임스텝 × 인코더+디코더를 고려하면, Full Attention은 기존 Block+Grid 대비 ~12배 연산 (halo 6,6 기준). 그러나 본 프로젝트는 **Full Attention 채택** (`05_symmetry_mode.md`, `2026-04-07 인코더 미결정 항목 및 Attention 가중치 비교.md` 참조) — 그 이유는 attention sink, halo cell의 boundary 정합성, 모든 cell 쌍 참조의 물리적 필요성 때문. 절대 FLOPs (~1.8 GFLOPs) 는 현대 GPU에서 trivial.

### 기타 구조적 경량화 방안

| 방향 | 절감 효과 | 정밀도 영향 | 비고 |
|------|:---:|:---:|------|
| **Block+Grid Attention** (Full 대비) | ~6× attention 절감 | 없음 | 기존 검증됨 |
| **Stem → Linear proj** | Stem 파라미터 89% 절감 (5.6M→0.7M) | 없음 | ViT patch embedding과 동일 패턴 |
| **Stage 수 조절** (3→2) | ~33% 절감 | 표현력 감소 가능 | 실험으로 판단 |
| **D_latent 조절** (128→64) | ~50% Mamba 연산 절감 | 정보 병목 가능 | 기존 64로 시작 |
| **시퀀스 길이** (576→288, Δt=10min) | 50% 전체 절감 | 시간 해상도 감소 | 물리적 타당성 검토 필요 |

## 5. 결론

| 항목 | 결정 |
|------|------|
| 학습 정밀도 | **FP32 기본** (TF32 자동 가속 포함) |
| Mixed precision | 초기 미도입. 속도 병목 시 AMP + physics loss FP32 유지 패턴 검토 |
| INT8/FP16 추론 | **미적용** (회귀 정밀도 위험, 핵 분야 사례 없음) |
| 경량화 | **구조적 효율화**: Block+Grid 유지, Stem 경량화, D_latent/stage 수 조절 |
