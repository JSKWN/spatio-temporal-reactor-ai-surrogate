# RT-2 — Vision-Language-Action: 행동을 토큰으로

> **출처**: Zitkovich et al. (2023). *RT-2: Vision-Language-Action Models Transfer Web Knowledge to Robotic Control.* [arXiv:2307.15818](https://arxiv.org/abs/2307.15818)
> **공개 수준**: 부분공개 (논문 공개, 모델 비공개)
> **우리 아키텍처 맥락**: **행동의 토큰화(action tokenization)** 및 **웹 지식 전이** 개념

---

## 1. 시스템 개요

### 직관

RT-2의 핵심 아이디어는 놀랍도록 단순하다: **로봇 행동을 텍스트 토큰으로 표현**한다. 7-DoF 관절 명령의 각 차원을 256개 구간으로 이산화하고, VLM의 가장 덜 사용되는 256개 토큰으로 매핑한다. 그러면 VLM이 "텍스트를 생성하듯" 행동을 생성한다.

### 아키텍처

```
[이미지 + "컵을 집어라"] → [PaLI-X 55B 또는 PaLM-E 12B (변경 없음)]
    → 자기회귀 텍스트 생성: "1 128 91 241 5 101 127 217"
    → 256-bin 역변환: 7-DoF 연속 행동 + 종료 플래그
```

**핵심**: 아키텍처를 전혀 수정하지 않음. 데이터 표현만 변경.

---

## 2. 기능별 상세 분석

### 2.1 행동 토큰화 (Action Tokenization)

```
7-DoF: [x, y, z, rx, ry, rz, gripper]
각 차원 → 256 bins으로 이산화 (= 약 0.4% 해상도)
예: x=0.523 → bin 134, y=-0.112 → bin 55, ...
출력: "134 55 201 128 64 192 240 0"   (8 tokens, 마지막은 종료)
```

**우리 문제에서의 시사점**: 우리 10ch 물리량 출력을 "토큰 시퀀스"로 표현하는 것은 이론적으로 가능하지만 **부적합**:
- 10ch × 720 cell = 7,200개 연속값을 이산화하면 정밀도 손실이 심각
- 물리량은 연속적이며, 이산화된 표현보다 직접 예측이 자연스러움
- RT-2 자체도 1-3Hz의 낮은 추론 속도가 한계로 지적됨

### 2.2 웹 지식 전이 (Co-fine-tuning)

```
학습 배치 구성:
  RT-2-PaLI-X: 50% 로봇 데이터 + 50% 웹 VQA/캡셔닝 데이터
  RT-2-PaLM-E: 66% 로봇 데이터 + 34% 웹 데이터
```

**결과**: 웹 데이터로 학습한 "일반 지식"이 로봇 제어에 전이. 학습에서 본 적 없는 물체도 조작 가능.

**우리 문제에서의 시사점**: "일반 물리 지식"을 대규모 물리 시뮬레이션 데이터에서 사전학습하고, 원자로 특화 데이터로 미세조정하는 전략. 그러나 현재 우리 데이터(100 LP × 20 프로파일)가 이미 도메인 특화이므로, 웹 스케일 사전학습의 이점은 제한적. NVIDIA Modulus 등 물리 사전학습 프레임워크와 더 관련.

### 2.3 Chain-of-Thought 행동 생성

RT-2의 변형: 행동 토큰 전에 **자연어 "계획" 단계**를 출력:
```
입력: [이미지] + "컵에 물을 따라라"
출력: "Plan: 주전자를 찾아 집은 후 컵 위로 기울인다. Action: 134 55 ..."
```

**우리 문제에서의 시사점**: 해석 가능성 관점에서 흥미롭지만, 우리 모델은 물리량을 직접 예측하므로 언어적 "계획" 단계가 불필요.

## 3. 우리 문제 적용성 평가

| 기법 | 적용 가능 | 근거 |
|------|----------|------|
| 행동 토큰화 (256-bin) | X | 7,200개 연속값 이산화 시 정밀도 손실 심각 |
| 웹 지식 전이 | △ | 물리 사전학습과 관련. 현재 도메인 특화 데이터로 충분 |
| Chain-of-Thought | X | 물리량 직접 예측에 언어 계획 불필요 |
| **아키텍처 무수정 원칙** | △ | "데이터 표현을 바꿔서 기존 모델을 재활용"하는 철학은 참고 가치 |

### 핵심 차용
- **설계 철학**: "아키텍처를 복잡하게 만들지 말고, 데이터 표현을 맞추라." 우리도 인코더 입력(21ch)의 정규화/표현을 최적화하는 것이 아키텍처 변경보다 효과적일 수 있음
