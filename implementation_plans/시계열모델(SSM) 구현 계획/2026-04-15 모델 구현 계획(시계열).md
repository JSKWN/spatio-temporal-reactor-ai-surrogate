## 3. 데이터 입출력 형태

### 3.1 공간 인코더 입력 (이미 구현됨)

```
입력: (B, 20, 6, 6, **23, 2026-04-22 갱신**)
  - 20 axial layers
  - 6×6 quarter-core halo grid (5×5 연료집합체 + 1 halo)
  - **23ch = state(12) + xs_fuel(10) + rod_map(1)**
    - **state 12ch (2026-04-22 신설 Σ_f_g1/g2 포함)**: power, T_cool, T_fuel, ρ_cool, N_Xe, N_I, φ_g1, φ_g2, σ_a^Xe_g1, σ_a^Xe_g2, **Σ_f_g1, Σ_f_g2 (신설)** (정규화됨)
    - xs_fuel 10ch: 시나리오별 고정 거시단면적 (정규화됨)
    - rod_map 1ch: 제어봉 삽입 깊이 공간 분포 (minmax 정규화)

p_load: 스칼라 (minmax 정규화), 매 시점 부하 출력 비율
```

### 3.2 인코더 출력 → 프로세서 입력

```
인코더 출력: (B, 20, 6, 6, 128)
    ↓ reshape
프로세서 입력: (B, 720, 128)
  - 720 = 20 × 6 × 6 cell의 평탄화된 토큰 시퀀스
  - 각 cell이 하나의 토큰 (TokenWM 패턴)
  - 128D 잠재 벡터
```

### 3.3 시간 축 처리

CRS 학습 시:
```
입력: (B, T, 720, 128) — T = 575 step의 시퀀스
  - Mamba parallel scan으로 시간 축을 한번에 처리
  - 각 시점에 p_load(t) 스칼라 concat
```

추론 시:
```
입력: (B, 720, 128) — 1 step씩
  - InferenceParams로 hidden state 캐시
  - 자기회귀 rollout
```

## 5. 시공간 프로세서 (Mamba-3 + Attention 하이브리드)

### 5.1 전체 구조

> **단위 명명**: 우리 프로세서는 `[Mamba, Mamba, Attention+MLP]` 구성을 1개 **Block**으로 정의. **N=3 Blocks** = 총 Mamba 6개 + Attention 3개 + MLP 3개의 sub-module (baseline N=3 확정, 2026-04-20).

```
입력: z(t) ∈ (B, 720, 128) + p_load 스칼라
    ↓
┌─────────────────────────────────────────┐
│  Block 1                                 │
│  ├── Mamba-3 #1                          │
│  ├── Mamba-3 #2                          │
│  └── Self-Attention #1 + 소형 MLP        │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│  Block 2                                 │
│  ├── Mamba-3 #3                          │
│  ├── Mamba-3 #4                          │
│  └── Self-Attention #2 + 소형 MLP        │
└─────────────────────────────────────────┘
    ↓
출력: d(t) ∈ (B, 720, 128)
```

- **Block 1, 2는 완전 독립 가중치** (가중치 공유 없음)
- 각 Block 내부에서 Mamba → Mamba → Attention+MLP 순서로 데이터 흐름
- **Mamba 뒤에는 MLP 없음**. **Attention 뒤에만 소형 MLP 부착**

### 5.2 Block 내부 흐름

각 Block 1개의 데이터 흐름:

```
입력 z ∈ (B, 720, 128) + p_load
    ↓
[Mamba-3 #1] (cell별 독립 시간 전이)
    ├── 입력: z + p_load concat
    ├── 시간 축으로 parallel scan (학습) 또는 1-step (추론)
    └── 출력: m1(t) ∈ (B, 720, 128)
    ↓
[Mamba-3 #2] (cell별 독립 시간 전이, 한번 더)
    └── 출력: m2(t) ∈ (B, 720, 128)
    ↓
[Self-Attention] (각 시점 t에서 720 cell 간 공간 결합)
    ├── 입력: m2(t) (시점별 독립 처리, 시간 축 stateless)
    └── 출력: a_pre(t) ∈ (B, 720, 128)
    ↓
[소형 MLP/FFN] (channel mixing, 비선형성)
    ├── 입력: a_pre(t)
    ├── hidden 확장 비율: 2× 또는 4× (튜닝 단계 결정)
    └── 출력: a(t) ∈ (B, 720, 128)
    ↓
출력: a(t)
```

### 5.3 Mamba-3 블록 상세

| 항목 | 값 | 비고 |
|------|---|------|
| hidden dim (d_model) | 128 | 인코더 출력 차원 일치 |
| state_dim (N) | **32** | Drama 표준 16 → 확장 (2026-04-20 — 다중 동역학 중첩 수용) |
| head_dim | 64 | Mamba-2 SSD 표준 |
| num_heads | 2 | 128/64 |
| 복소 상태 | **ON** | Mamba-3 핵심. Xe 진동 위상 추적 |
| Trapezoidal 이산화 | **ON** | Mamba-3 핵심. Euler 대비 O(Δ³) 정밀도 |
| Conv1D | **OFF** | Trapezoidal이 인접 시점 정보 자동 포함 |
| 학습 모드 | parallel scan | 575 step 전체 한번에 |
| 추론 모드 | 1-step recurrence | InferenceParams 캐시 |

### 5.4 Attention 블록 상세

| 항목 | 값 | 비고 |
|------|---|------|
| Type | Multi-Head Self-Attention | 표준 Transformer attention |
| num_heads | 4 | 128 / 32 (head_dim=32) |
| head_dim | 32 | |
| 적용 범위 | 720 cell (각 시점 독립) | stateless |
| 시점당 복잡도 | O(720²) ≈ 518K | 인코더 FullAttn과 동일 규모 |

### 5.5 p_load 주입 위치

- **Mamba-3 입력에 concat**: 매 시점 p_load(t)를 z(t)와 concat 후 Mamba 입력
- Attention 블록에는 별도 주입 없음 (Mamba를 통해 간접 전달)

### 5.6 순서 결정 근거 (Mamba → Attention)

- 인코더가 입력 시점 t의 공간 결합을 이미 수행 → z(t)에 공간 정보 포함
- Mamba가 먼저 cell별 시간 전이 → 시간 누적 정보 형성
- Attention이 시간 전이 후 결과의 공간 일관성을 매 시점 재정리
- Jamba 패턴 (LLM에서 검증된 학습 안정성)

## 6. 디코더 (Conv3D + 잔차 결합)

### 6.1 전체 흐름

```
프로세서 출력: d(t) ∈ (B, 720, 128)
    ↓ reshape
(B, 20, 6, 6, 128)
    ↓
[Neural Decoder: Conv3D 1×1×1]  (학습 가능)
  Channel: 128 → 10
    ↓
delta_pred ∈ (B, 20, 6, 6, 10)  (정규화된 공간의 잔차)
    ↓
[잔차 결합 (텐서 덧셈)]  ← 학습 가중치 없음, 단순 함수
  X_next_pred = state(t)[**12ch**] + delta_pred
    ↓
X_next_pred ∈ (B, 20, 6, 6, **12**)  (정규화된 절대값, 2026-04-22 갱신)
    ↓
[Loss 계산] (다음 섹션 참조)
```

### 6.2 Neural Decoder (Conv3D 1×1×1)

| 항목 | 값 | 비고 |
|------|---|------|
| Type | Conv3D, kernel=(1,1,1) | cell별 독립 변환. spatial dim (D, H, W, C) 명시적 유지 |
| 입력 채널 | 128 | 프로세서 출력 |
| 출력 채널 | **12 (2026-04-22 갱신)** | 물리량 채널 수 (state 12ch에 대응, Σ_f_g1/g2 신설) |
| 활성화 | (튜닝 단계 결정) | 잠정: 마지막은 활성화 없음 (선형 출력) |
| 깊이 확장 가능성 | 필요 시 다층으로 확장 가능 | 시작은 1층 |

**선택 근거**:
- spatial dim (D, H, W, C) 명시적 유지 → Physical Loss 계산 시 텐서 형태가 자연스러움
- 경계 조건 (halo cell) 처리 용이
- 사용자 직관에 부합

### 6.3 잔차 결합 (단순 함수)

학습 가능한 가중치 **없음**. 단순한 두 가지 연산:

```python
# 1. 정규화 공간에서 덧셈
X_next_pred = state(t)[10ch] + delta_pred  # 둘 다 정규화된 텐서

# 2. (Loss 계산 시) 역정규화하여 물리 단위로 복원
X_next_pred_physical = denormalize(X_next_pred)
state_t_physical = denormalize(state(t)[10ch])
```

**`state(t)`의 출처 명확화**:
- `state(t)`는 **인코더 입력 21ch 중 state 부분 10ch** (Xe, I, flux, 온도 등 물리량)
- xs_fuel(10ch), rod_map(1ch)은 잔차 결합에서 사용 안 함 (예측 대상이 아님)
- 정규화된 절대값
- **학습 시**: 데이터셋에서 직접 로드 (해당 시점의 GT 입력)
- **자기회귀 rollout 시**: 이전 step의 예측값 X_next_pred (직전 시점 디코더 출력)

`tf.keras.layers.Layer`로 만들 수도 있고 그냥 함수로 둬도 무방. 학습 파라미터가 없으므로 사실상 후처리 단계.

### 6.4 잔차 예측의 의의

- **정출력 구간** (5분 동안 거의 변화 없음): delta ≈ 0 → 신경망이 0 출력하는 것은 쉬움
- **변동 구간**: 미세 변화량만 예측 → 신경망이 큰 절대값을 매번 정확히 모사할 필요 없음
- **누적 오차 제한**: 절대값 직접 예측 시 항등 함수 학습 부담 회피
- GraphCast (Science 2023) 등 물리 AI의 표준 패턴

## 7. Loss 적용 위치 + 데이터 형태

### 7.1 핵심 구조 요약

| 구성 요소 | 형태 |
|---------|------|
| **모델 출력** | **잔차 (delta_pred)** — Conv3D 1×1×1의 출력. 잔차 예측의 학습 효율성 |
| **L_data** | **정규화 공간에서 절대값 비교** = MSE(state(t)+delta_pred, X_next_gt) |
| **L_physics** | **역정규화된 물리 단위에서 계산** (확산 잔차, Bateman ODE, L_σXe) |

> **참고**: `MSE(state(t)+delta_pred, X_next_gt)`와 `MSE(delta_pred, delta_gt)`는 **수학적으로 동치** (state(t)가 입력 상수라 elementwise 차분 동일). 본 문서에서는 가독성을 위해 **절대값 비교 형태로 표기**.

### 7.2 데이터 형태별 사용처

| 형태 | 정의 | 어디서 사용 |
|------|------|-----------|
| ① **정규화된 절대값** | 데이터셋의 표준 형태 (각 채널별 z/log_z/minmax 정규화) | 인코더 입력, 잔차 결합의 base, L_data 비교 |
| ② **물리 단위 절대값** | 정규화 절대값을 역정규화 → 실제 물리 단위 (n/cm²/s, ppm, °C 등) | L_physics 계산 |

> **참고**: "정규화된 잔차(delta_pred)"는 모델 내부 중간 결과 (Conv3D 출력)이며 **별도 loss target이 아님** (위 §7.1 참고).

### 7.3 데이터 흐름 다이어그램

```
[데이터셋 로드]
  state_t_normalized      X_next_gt_normalized            ← ① 정규화 절대값
       ↓                          ↓
[인코더 → 프로세서 → Conv3D]
       ↓
  delta_pred_normalized                                    ← (모델 내부 중간 출력)
       ↓
[잔차 결합]
  X_next_pred_normalized = state_t_normalized + delta_pred_normalized   ← ① 정규화 절대값
       ↓
       ├──────────────────────────────────────┐
       ↓                                       ↓
[L_data 계산 — 정규화 공간]              [역정규화]
  L_data = MSE(                          state_t_physical = denormalize(state_t_normalized)
    X_next_pred_normalized,              X_next_pred_physical = denormalize(X_next_pred_normalized)
    X_next_gt_normalized                                                       ↓ ② 물리 단위
  )                                       [L_physics 계산 — 물리 단위 공간]
                                            ├── L_diff_rel: 인접 cell 간 확산 잔차 (X_next_pred_physical 사용)
                                            ├── L_Bateman: Xe/I ODE 잔차 (state_t_physical, X_next_pred_physical 사용)
                                            └── L_σXe: Taylor 단면적 정합성
```

### 7.4 핵심 요점

- **잔차 예측의 효과**는 **모델 출력 형태 (Conv3D가 작은 변화량만 출력)**에서 옴. Loss 표기 형태와 무관하게 동일한 학습 효과
- **Physical Loss를 정규화 공간이 아닌 물리 단위에서 계산하는 이유**:
  - 확산 방정식, Bateman ODE는 **단위 있는 물리량**에서 정의됨
  - 정규화 공간에서는 ∇²φ, σ_a·φ 같은 항의 차원이 맞지 않음
- **잔차 자체에 Physical Loss 적용 불가**:
  - delta는 변화량이지 절대값이 아니므로 물리 연산자 적용 불가
- **역정규화의 미분 가능성**:
  - 우리 정규화 방식 (zscore, log_zscore, minmax, pcm_zscore) 모두 미분 가능
  - gradient가 역정규화를 통과하여 Conv3D / 프로세서 / 인코더까지 손실 없이 흐름

## 9. 추론 프로토콜

### 9.1 모델의 본질 — 1-step 예측 함수

모델 내부에는 별도의 "CRS 모드"나 "Branch 모드"가 **없음**. 모델은 단순히 매 시점 query를 받아 다음 시점을 예측하는 함수:

```
f(state(t), rod_map_query, p_load_query, h(t))  →  state(t+1)_pred,  h(t+1)
```

다른 rod_map_query → 다른 예측. 모델은 query가 시뮬레이터의 임계 값인지 사용자가 임의로 만든 What-if 값인지 구분하지 않음.

CRS/Branch라는 구분은 **사용자가 결과를 어떻게 해석/활용하느냐의 문제**:
- 사용자가 시뮬레이터의 임계 rod_map을 query로 주면 → 결과는 "CRS 예측"
- 사용자가 임의의 다른 rod_map을 query로 주면 → 결과는 "Branch 예측" (What-if)

#### h(t)의 본질: 단일 텐서 (시점별 누적 압축)

Mamba의 hidden state는 **시점별로 누적되어 압축된 단일 텐서**:

```
h(0) = zeros (초기)
h(1) = Mamba_step(z(1), h(0))   ← h(1) 안에 t=1까지의 정보 압축
h(2) = Mamba_step(z(2), h(1))   ← h(2) 안에 t=1,2 정보 모두 압축
h(3) = Mamba_step(z(3), h(2))   ← h(3) 안에 t=1,2,3 정보 모두 압축
```

T step 진행 후 모델이 "기억"하는 것은 **h(T) 텐서 1개**. 과거 h(1), h(2), …, h(T-1)을 별도로 보관하지 않음 — 모든 과거 정보가 h(T)에 압축되어 있으므로 불필요.

이는 GRU/LSTM/Mamba 같은 모든 recurrent 모델의 본질적 동작.

### 9.2 1-step 예측 (단발 query)

```
입력:
  state(t)        ∈ (B, 20, 6, 6, 10) — 노심 상태 (정규화)
  rod_map_query   ∈ (B, 20, 6, 6, 1)  — 사용자 지정 제어봉 위치
  p_load_query    ∈ (B,)               — 사용자 지정 부하 출력
  h(t)            — Mamba hidden state (initial: zeros, 또는 이전 누적 상태)

처리:
  1. z(t) = Encoder(state(t), rod_map_query, p_load_query)  ← 인코더가 21ch 입력 처리
  2. d(t), h(t+1) = Processor(z(t), p_load_query, h(t))     ← 프로세서 1-step (Mamba state 갱신)
  3. delta_pred = Conv3D(d(t))                              ← 디코더
  4. state(t+1)_pred = state(t)[10ch] + delta_pred          ← 잔차 결합

출력:
  state(t+1)_pred
  h(t+1)
```

### 9.3 사용자 측 운영 패턴

모델은 §9.2의 1-step 함수만 제공. 사용자가 어떻게 query를 구성하느냐에 따라 운영 양상이 달라짐.

#### 패턴 A: 연속 자기회귀 (시뮬레이터 재현 등)

사용자가 시점별 query를 순차 입력. 매 step의 h(t)를 그대로 다음 query에 전달.

```
초기: state(0), h(0) = zeros (또는 사용자 지정)

for t in 0..T-1:
    # 사용자가 t+1 시점의 query 입력 (예: 시뮬레이터 CRS rod_map)
    rod_map_query, p_load_query  ← 외부에서 주어짐

    # 1-step 예측 (§9.2)
    state(t+1)_pred, h(t+1) = Model(state(t), rod_map_query, p_load_query, h(t))

    # 다음 step의 입력으로 자신의 예측 사용
    state(t) ← state(t+1)_pred
    h(t)     ← h(t+1)  (그대로 누적)
```

#### 패턴 B: 동일 시점 다중 query (What-if 평가)

같은 h(t), 같은 state(t)에 대해 여러 rod_map_query를 시도. 각각 독립적인 1-step 예측.

```
주어진 시점 t에서:
  state(t), h(t) 고정

  # 핵심: Mamba step은 h(t)를 갱신하므로 (h(t) → h(t+1)),
  # 같은 h(t)로 여러 query를 시도하려면 미리 snapshot 필요
  h_backup = deepcopy(h(t))

  for rod_map_query in 다중_후보:
      state(t+1)_pred, _ = Model(state(t), rod_map_query, p_load_query, h_backup)
      # 매 호출마다 h_backup의 동일 사본 사용 (Model이 hidden state 사본 받음)
      results.append(state(t+1)_pred)
```

#### 패턴 C: 자기회귀 + 분기 평가 혼합

자기회귀 진행 중 특정 시점에서 What-if도 함께 평가. 분기 평가 후 원래 자기회귀를 계속하려면 h(t) 보존 필수.

```
연속 자기회귀 진행 중 시점 t에서:

1. h_backup = deepcopy(h(t))                        ← snapshot

2. # 분기 평가 (패턴 B와 동일)
   for rod_map_query in 다중_후보:
       branch_pred, _ = Model(state(t), rod_map_query, p_load_query, h_backup)
       branch_results.append(branch_pred)

3. # 원래 자기회귀 계속 (실제 사용할 query로 진짜 1-step 진행)
   state(t+1), h(t+1) = Model(state(t), rod_map_main, p_load_main, h_backup)

4. 다음 시점 자기회귀 (h(t+1) 사용)
```

#### 패턴 D: Iterative Critical Search (수동 CRS 탐색)

매 시점마다 사용자가 keff=1 만족하는 rod_map을 탐색하며 자기회귀로 진행. 패턴 C의 특수 사례 — **분기 평가가 자기회귀의 일부**로 기능.

```
초기: state(0), h(0) = zeros

for t in 0..T-1:
    h_backup = deepcopy(h(t))                       ← 매 시점 snapshot

    # 탐색 루프: keff=1 만족하는 query를 찾을 때까지 반복
    while True:
        rod_map_query, p_load_query  ← 사용자가 후보 지정 (수동 또는 자동 탐색기)

        state(t+1)_pred, h(t+1)_candidate = Model(
            state(t), rod_map_query, p_load_query, h_backup
        )

        keff_pred = state(t+1)_pred에서 추출 (또는 별도 head)

        if abs(keff_pred - 1.0) < tolerance:
            break       ← 채택, 탐색 종료
        else:
            continue    ← 다음 후보 시도 (h_backup은 그대로 유지)

    # 채택된 query의 결과를 commit
    state(t+1) ← state(t+1)_pred
    h(t+1)    ← h(t+1)_candidate

    # 다음 시점으로 진행
    state(t) ← state(t+1)
    h(t)     ← h(t+1)
```

**의미**:
- 모델은 "임계 조건 만족 여부"를 직접 판단하지 않음. 사용자가 결과의 keff를 보고 채택/거부 결정
- 거부된 query의 h(t+1)_candidate는 폐기 (h_backup 보존되므로 영향 없음)
- 채택된 시점에서만 h를 commit하여 누적
- 시뮬레이터의 임계 탐색 알고리즘을 **모델 추론 루프 안에서 재현**하는 패턴

### 9.4 핵심 요점

- 모델은 **단순한 1-step 함수**. CRS/Branch는 사용자 query 구성/해석의 문제
- **h(t) snapshot/restore는 메모리 관리 절차**, 모델 내부 모드가 아님
- Mamba step은 h(t)를 갱신하므로, 동일 h(t)로 여러 query를 시도하려면 snapshot 필요
- Attention 블록은 stateless라 별도 snapshot 불필요
